## 概要

> フィーチャーブランチで開発したアプリコード（`backend/`・`frontend/`）が master にコミットされていない場合、master にチェックアウトしても当該ディレクトリは未追跡ファイルとして作業ディレクトリに残り続ける。意図せず `git add` してしまうリスクがあるため、明示的に対処が必要。

---

## 背景・コンテキスト

**状況:** feature/dev-flow-test ブランチで Todoアプリを開発し、汎用資産のみ master にマージした後、master ブランチに切り替えた。

**課題・問題:** `backend/` と `frontend/` が master の `.gitignore` に含まれておらず、また master にコミットもされていないため、`git status` に未追跡ファイルとして表示され続けた。

---

## 事実（何が起きたか / 何を決めたか）

- git はコミットされていないファイルはブランチ切り替え時に削除しない（未追跡ファイルはどのブランチにも属さない）
- `feature/dev-flow-test` でのアプリコードが master 上に未追跡ファイルとして残った
- 対処として対象ディレクトリを削除（`rm -rf backend/ frontend/`）した

---

## 理由・分析

### 検討した選択肢

| 選択肢 | メリット | デメリット | 採用 |
|--------|---------|----------|------|
| ディレクトリを削除する | master が完全にクリーンになる | フィーチャーブランチに戻ると再クローン or checkout が必要 | ✅ 採用 |
| `.gitignore` に追加する | git status に出なくなる | ファイルは残り続け、容量を占有する | ❌ 却下 |
| そのままにする | 手間がかからない | 誤って git add するリスクがある | ❌ 却下 |

---

## 次回への示唆

### すべきこと（Do）

- [x] フィーチャーブランチの作業完了後に master へ戻る前に、アプリコードの扱いを決めておく（削除 or gitignore）
- [x] master を「フレームワークのみ」の状態として管理する場合、アプリコードディレクトリは `rm -rf` で削除してからコミット作業を締める
- [x] master に切り替えた直後に `git status` で未追跡ファイルがないか確認する

### 避けること（Don't）

- master 上で `git add .` や `git add -A` を使わない（意図しないアプリコードを巻き込むリスク）
- 未追跡ファイルを「害がないから」と放置しない（次の開発者が混乱する）
