# ナレッジベース（knowledge）

開発サイクルを回すたびに得られた知識・経験を蓄積し、次のサイクルへ活かすための仕組みです。  
「同じ失敗を繰り返さない」「良いパターンをチームで共有する」ことを目的としています。

---

## ディレクトリ構成

```
knowledge/                    # ナレッジの蓄積先（ここ）
├── INSTRUCTIONS.md
├── design-decisions/
├── review-findings/
├── test-patterns/
└── lessons-learned/

rules/knowledge/              # ナレッジ記録テンプレート
└── template.md
```

---

## いつ記録するか

| タイミング | 記録内容 | 保存先 |
|-----------|---------|--------|
| 詳細設計レビュー後 | 設計判断の理由・却下した選択肢 | `design-decisions/` |
| コードレビュー後 | 指摘されたパターン・改善観点 | `review-findings/` |
| テスト実施後 | 発見したバグのパターン・再発防止策 | `test-patterns/` |
| フェーズ完了後 | フェーズ全体の振り返り・教訓 | `lessons-learned/` |
| 重大インシデント後 | 原因・対応・予防策 | `lessons-learned/` |

### 記録の判断基準

以下のどれか1つに当てはまれば記録する価値がある。

- 「次回また同じことで迷いそう」と感じた判断
- 「これは気づきにくい」と感じたバグや指摘
- 「このやり方は効果的だった」と感じたテスト・設計手法
- チーム内で議論になった決定事項

---

## 何を記録するか

### 記録すべきもの

- **設計判断の理由**: なぜそのアーキテクチャを選んだか、何を却下したか
- **繰り返し起きるパターン**: 同じ指摘が複数のPRで出た場合
- **非自明なバグの根本原因**: 調査に時間がかかったバグの仕組みと発見方法
- **効果的だったテスト手法**: カバレッジより品質が上がったアプローチ
- **コスト・パフォーマンスのトレードオフ**: 速度 vs. 保守性などの選択根拠

### 記録しないもの

- 公式ドキュメントに書いてあること
- コードを読めば自明な内容
- 一時的な作業メモ（Issueやタスク管理で扱う）

---

## どう記録するか

### ファイル命名規則

```
YYYYMMDD_短いタイトル.md
例: 20260414_statemanagement-zustand-vs-context.md
```

### 記録フォーマット

`docs/rules/knowledge/template.md` をコピーして使用する。  
**最低限、以下の3点を書くこと:**

1. **何が起きたか / 何を決めたか** （事実）
2. **なぜそうなったか / なぜそう決めたか** （理由）
3. **次回どうすべきか** （示唆）

## 活用のルール

### 検索して使う

新しいフェーズを始める前に、関連するナレッジを検索する。

```bash
# キーワードで検索
grep -r "キーワード" docs/knowledge/

# カテゴリで絞り込む
ls docs/knowledge/design-decisions/
```

### 定期的に見直す

- **スプリント/イテレーション終了時**: `lessons-learned/` に追記
- **四半期ごと**: 古くなったナレッジを `[OUTDATED]` タグをつけてアーカイブ
- **新メンバー参加時**: `design-decisions/` を読み合わせのインプットに使う

### 更新・修正

ナレッジが誤っていたり古くなっていた場合は、新しいエントリを追加するのではなく、  
元のファイルを直接更新し、ファイル内の「最終更新日」を変更する。

## カテゴリ別ガイド

| カテゴリ | 説明 | テンプレートの使い方 |
|---------|------|-----------------|
| [design-decisions/](design-decisions/INSTRUCTIONS.md) | 設計判断とその根拠。却下した選択肢も記録する | `context` と `decision` セクションを重点的に |

| [lessons-learned/](lessons-learned/INSTRUCTIONS.md) | フェーズ・プロジェクト全体の振り返り | `what-went-well` と `what-to-improve` を対で書く |
| [reflected/](reflected/INSTRUCTIONS.md) | `/reflect-knowledge` でルールに反映済みのナレッジ | 参照用。編集する場合は元のカテゴリに戻してから |
