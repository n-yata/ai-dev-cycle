# 設計判断（design-decisions）

過去の設計判断とその理由を記録するカテゴリです。  
「なぜこの設計にしたのか」「なぜ他の選択肢を却下したのか」を残すことで、  
同じ議論を繰り返さず、将来の変更判断の根拠にできます。

---

## このカテゴリに記録するもの

- アーキテクチャの選択（例: 状態管理ライブラリの選定、DBスキーマ設計）
- 技術スタックの決定（例: フレームワーク、ORM、テストツールの選定）
- 設計パターンの採用（例: レイヤードアーキテクチャ vs. オニオンアーキテクチャ）
- 非機能要件のトレードオフ（例: 速度 vs. 保守性、シンプルさ vs. 拡張性）
- 意図的に「やらないこと」にした判断

---

## 記録のポイント

設計判断ナレッジで最も重要なのは **「却下した選択肢と理由」** です。  
採用した理由だけでなく、何を捨てたかを残してください。

```markdown
## 検討した選択肢

| 選択肢 | メリット | デメリット | 採用 |
|--------|---------|----------|------|
| Zustand | 軽量、ボイラープレート少 | デバッグツールが弱い | ✅ 採用 |
| Redux Toolkit | 強力なデバッグ、実績 | 学習コスト、冗長 | ❌ 却下 |
| Context API | 追加ライブラリ不要 | 大規模では再レンダリング問題 | ❌ 却下 |
```
