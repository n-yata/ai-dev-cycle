## 1. 背景・課題

現在、CloudFront + API Gateway + Lambda 構成で公開する業務 API に対して、以下の2種類の呼び出し元が存在する。

- **外部 SaaS（ServiceNow）**: 社外ネットワークからのAPI呼び出し
- **内部バックエンドサーバー**: AWS 内部からのAPI呼び出し

現状の課題:

- 呼び出し元ごとに異なるセキュリティ要件（外部は厳格な認証・認可、内部は信頼ベースの簡略化）があるが、統一的なアーキテクチャが未整備
- API の改ざん検知メカニズムが未実装であり、リクエストの完全性を保証できない
- ロールベースのアクセス制御がなく、API 単位での権限管理ができていない

---

## 2. ゴール・目的

- **統一認証基盤の確立**: 外部 SaaS・内部バックエンドの両方からの呼び出しを、単一のアーキテクチャで安全に受け付ける
- **改ざん検知の実現**: リクエストの署名検証により、通信経路上での改ざんを検知・拒否する
- **ロールベース認可の導入**: ユーザーのロールに基づいて API 単位でアクセスを制御する
- **AWS 標準機能の最大活用**: 自前実装を最小化し、IAM 認証（SigV4）等の AWS マネージドサービスでセキュリティ要件を満たす

---

## 3. スコープ

### 対象範囲

- API Gateway の認証方式（IAM 認証 / SigV4）の設計・実装
- JWT（RS256）によるトークン発行・検証の仕組み
- ロール × API のアクセス制御（DynamoDB ベース）
- JWT スキップ用ホワイトリスト（内部 BE 向け）
- CloudFront + WAF によるエッジ防御
- ServiceNow 用・内部 BE 用の API Gateway リソース分離
- Secrets Manager による鍵管理

### 対象外

- 業務 API 自体のビジネスロジック設計・実装（認可基盤を通過した後の処理）
- フロントエンド（UI）の設計・実装
- ServiceNow 側のワークフロー設計・実装
- RDS 上のユーザーマスタ自体の管理機能（CRUD 画面等）
- 鍵ローテーションの自動化（初期は手動運用、将来対応）

## 5. 機能要件

| # | 機能名 | 概要 | 優先度 |
|---|--------|------|--------|
| F-001 | SigV4 による改ざん検知 | API Gateway の IAM 認証により、リクエストヘッダ・ボディの署名検証を実施。改ざんされたリクエストを自動拒否する | Must |
| F-002 | JWT トークン発行 | ユーザー ID を受け取り、RDS からロール ID リストを取得し、RS256 署名付き JWT を発行する Lambda | Must |
| F-003 | JWT 署名検証 | ビジネス Lambda 内で JWT の RS256 署名を検証し、不正・期限切れトークンを拒否する | Must |
| F-004 | ロール × API アクセス制御 | JWT 内のロール情報と DynamoDB のロール × API 紐づけテーブルを照合し、API 単位で認可判定を行う | Must |
| F-005 | JWT スキップ ホワイトリスト | 内部 BE サーバーからの呼び出しは JWT 認可をスキップし、SigV4 認証のみで通過させる | Must |
| F-006 | API Gateway リソース分離 | ServiceNow 用（API Gateway A）と内部 BE 用（API Gateway B）を分離し、WAF ルール・スロットリングを個別設定可能にする | Must |
| F-007 | エッジ防御（CloudFront + WAF） | IP 制限、レート制限、SQLi/XSS 対策を CloudFront + WAF で実現する | Must |
| F-008 | シークレット管理 | RSA 鍵ペアを Secrets Manager で管理し、Lambda から安全に取得する | Must |

## 6. 非機能要件

### パフォーマンス

- JWT 検証を含むビジネス Lambda の認可処理: 100ms 以内（コールドスタート除く）
- DynamoDB からのロール × API 照合: 10ms 以内（単一 GetItem）
- JWT 発行 Lambda のレスポンス: 500ms 以内（RDS + Secrets Manager アクセス含む）

### セキュリティ

- すべての API 呼び出しは SigV4 署名による改ざん検知を必須とする
- JWT 署名方式は RS256（非対称鍵）を採用し、秘密鍵と公開鍵を分離管理する
- IAM ポリシーは最小権限の原則に従い、各 Lambda・各呼び出し元に必要最小限の権限のみ付与する
- ソースコード内への AWS アカウント ID・URL・キー等のハードコードを禁止する
- CloudTrail による API 呼び出しの監査ログを有効化する
- OWASP API Security Top 10 の観点でセキュリティレビューを実施する

### 可用性

- API Gateway + Lambda のサーバーレス構成により、AWS リージョン単位の可用性（99.95% 以上）に準拠する
- DynamoDB はオンデマンドキャパシティで開始し、スロットリングを回避する

### 保守性

- インフラは IaC（CDK or Terraform）で管理し、手動コンソール操作を原則禁止する
- Lambda のログレベルは環境変数で制御可能とする
- ホワイトリスト・ロール × API の変更は IaC 経由の PR レビュー必須とする

### 拡張性

- 呼び出し元の追加（新たな外部 SaaS、別の内部サービス等）に対して、API Gateway + IAM ユーザー/ロールの追加で対応可能な設計とする
- ロール・API の追加は DynamoDB テーブルへのエントリ追加のみで完結する設計とする
- 鍵ローテーションは `kid` ヘッダによる新旧鍵並行運用を将来的にサポートする

## 7. 制約条件

### 技術的制約

- AWS 環境上に構築する（リージョン: ap-northeast-1）
- API Gateway の認証方式は IAM 認証（SigV4）を採用する
- JWT 署名は RS256（非対称鍵）とする
- 認可データストアは DynamoDB（シングルテーブルデザイン）を採用する
- ユーザー × ロール管理には既存の RDS（PostgreSQL）を利用する
- Lambda ランタイムは TypeScript（Node.js）を採用する
- ServiceNow からの呼び出しは SigV4 署名 + JWT Bearer トークンの2層認証とする
- 内部 BE サーバーからの呼び出しは SigV4 署名のみ（JWT 不要、ホワイトリストでスキップ）

### 予算・期間制約

- サーバーレス構成（Lambda + DynamoDB オンデマンド）により、従量課金でコストを最小化する
- 特になし（シャビと要相談）

### 法的・規制制約

- 特になし（シャビと要相談）

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| SigV4（Signature Version 4） | AWS の標準リクエスト署名プロトコル。リクエストヘッダ・ボディを秘密鍵で署名し、改ざん検知と送信者認証を行う |
| JWT（JSON Web Token） | JSON ベースのトークン形式。ユーザー情報やロール情報をペイロードに含み、署名で完全性を保証する |
| RS256 | RSA 秘密鍵で署名し、公開鍵で検証する JWT 署名アルゴリズム。非対称鍵方式 |
| ホワイトリスト | JWT 認可をスキップできる呼び出し元（API Gateway の API ID）の一覧。内部 BE 用 |
| ロール × API 紐づけ | どのロールがどの API（メソッド + パス）を呼び出せるかの権限定義 |
| API Gateway A | ServiceNow（外部 SaaS）専用の API Gateway リソース |
| API Gateway B | 内部バックエンドサーバー専用の API Gateway リソース |
| ビジネス Lambda | 認可チェック（ホワイトリスト → JWT 検証 → ロール確認）+ 業務処理を行う Lambda |
| JWT 発行 Lambda | ユーザー ID からロールを取得し、RS256 署名付き JWT を発行する Lambda |
| routeKey | API Gateway HTTP API のルート形式（例: `GET /orders/{id}`） |

## 9. リスクと対策

| # | リスク | 影響度 | 発生確率 | 対策 |
|---|-------|-------|---------|------|
| R-001 | ServiceNow 用 IAM アクセスキーの漏洩 | 高 | 低 | 最小権限ポリシー、定期ローテーション、CloudTrail 監視 |
| R-002 | RSA 秘密鍵の漏洩 | 高 | 低 | Secrets Manager で管理、JWT 発行 Lambda のみアクセス許可、CloudTrail 監査 |
| R-003 | ホワイトリスト誤登録による JWT バイパス | 高 | 低 | IaC 経由のみ変更可、PR レビュー必須、CloudTrail でテーブル変更監視 |
| R-004 | JWT 有効期限切れ・時刻ずれ | 中 | 中 | クライアント側トークンキャッシュ + 期限前再発行、許容クロックスキュー定義 |
| R-005 | ロール × API 設定漏れによる正常リクエスト拒否 | 中 | 中 | エントリ追加を IaC 化し PR レビュー、403 レート急増の CloudWatch 監視 |
| R-006 | WAF ルール過剰ブロック | 中 | 中 | カウントモードで事前運用、ServiceNow/内部 BE のパターンを事前検証 |
