# 教訓（lessons-learned）

各フェーズ・プロジェクトの振り返りから得た教訓を記録するカテゴリです。  
個別の技術的なパターンではなく、プロセス・コミュニケーション・意思決定など  
より広い視点での学びを蓄積します。

---

## このカテゴリに記録するもの

- フェーズ完了後の振り返り（KPT形式可）
- 予定より時間がかかった作業の原因と対策
- 「最初からこうしていればよかった」という設計・プロセスの気づき
- チームのコミュニケーションや意思決定に関する学び
- 重大インシデント・障害の事後分析（ポストモーテム）

---

## 記録のポイント

教訓の記録で最も重要なのは **「具体的な次のアクション」** です。  
「気をつける」「丁寧にやる」のような抽象的な教訓は価値がありません。

```markdown
# ❌ 悪い例（抽象的すぎる）
教訓: APIの設計は慎重に行う。

# ✅ 良い例（具体的なアクションがある）
教訓: BE設計レビュー前にFEの型定義を共有することで、
      レビュー後の手戻りを2件から0件に削減できた。

次回のアクション:
- 詳細設計テンプレートの「FE/BEインターフェース合意」セクションを必須化する
- 設計レビューのチェックリストに「型定義の共有済み確認」を追加する
```

## 振り返りのフレームワーク

### KPT形式

| 区分 | 説明 |
|------|------|
| **K**eep | うまくいったこと。次回も続けるべきこと |
| **P**roblem | うまくいかなかったこと。問題だったこと |
| **T**ry | 次回試したいこと。改善アクション |

### タイムライン形式（インシデント向け）

```
[日時] 事象発生
[日時] 検知
[日時] 原因特定
[日時] 対応開始
[日時] 解消確認
```
