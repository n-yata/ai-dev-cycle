# レビュー指摘パターン（review-findings）

コードレビューで繰り返し発見される指摘パターンを記録するカテゴリです。  
「同じ指摘を複数回しない」「新しいメンバーが同じミスをしない」ことを目的とします。

---

## このカテゴリに記録するもの

- 複数のPRで繰り返し出た指摘
- 一目では気づきにくいセキュリティ・パフォーマンスの問題
- 「良い例 / 悪い例」として共有する価値のあるコードパターン
- アーキテクチャの観点からの逸脱（レイヤー違反等）

---

## 記録のポイント

指摘パターンの記録で最も重要なのは **「具体的なビフォー/アフターコード」** です。  
抽象的な教訓より、コードで示すことを優先してください。

```typescript
// Bad: 指摘されたパターン（コメントで何が問題かを説明）
const handler = async (req, res) => {
  const user = await db.query(`SELECT * FROM users WHERE id = '${req.params.id}'`);
  // ^ SQLインジェクション脆弱性
};

// Good: 改善後のパターン
const handler = async (req, res) => {
  const user = await db.user.findUnique({ where: { id: req.params.id } });
  // ^ ORMのパラメータバインディングで安全
};
```

## よく出る指摘カテゴリ

| カテゴリ | 例 |
|---------|-----|
| セキュリティ | 入力検証漏れ、機密情報のログ出力、ハードコードされた認証情報 |
| パフォーマンス | N+1クエリ、不必要な再レンダリング、未使用のimport |
| 型安全性 | `any` の使用、型アサーションの乱用、null安全性の欠如 |
| テスタビリティ | 依存の密結合、副作用の混入、モックしにくい設計 |
| 可読性 | 長すぎる関数、意図が不明な変数名、コメントとコードの乖離 |
| エラー処理 | エラーの握りつぶし、ユーザーへのフィードバック不足 |
