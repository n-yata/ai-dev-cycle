# フェーズ０ 要件定義（requirements）

基本設計に入る前に、「何を作るか・なぜ作るか」を関係者全員で合意するフェーズです。  
機能要件・非機能要件・スコープを文書化し、基本設計（技術選定）の前提情報を整備することを目的とします。

---

## このフェーズの目的

- 解決すべき課題とゴールの明確化
- 機能要件・非機能要件の洗い出しと文書化
- スコープ（対象範囲・対象外）の合意形成
- 基本設計フェーズへの入力情報の整備

---

## 成果物一覧

| # | 成果物 | テンプレート | 単位 | 説明 |
|---|--------|-----------|------|------|
| 1 | 要件定義書 | [template.md](template.md) | アプリごと | 背景・ゴール・スコープ・機能要件・非機能要件・制約条件・リスク |

### ファイル命名規則

要件定義書は継続的に更新するため、ファイル名に日付は付けない。

```
docs/artifacts/requirements/{アプリ名}.md
例: docs/artifacts/requirements/todo-app.md
例: docs/artifacts/requirements/user-auth.md
```

---

## ディレクトリ構成

```
rules/requirements/              # ルール・テンプレート（ここ）
├── INSTRUCTIONS.md
└── template.md

artifacts/requirements/          # 成果物の出力先
└── {アプリ名}.md
```

---

## 要件定義ドキュメントの作成手順

1. `docs/rules/requirements/template.md` をコピーする
2. `docs/artifacts/requirements/` 配下に命名規則に従って保存する
3. テンプレートの各セクションを埋める
4. チームレビューを実施する
5. シャビの承認後、基本設計フェーズへ移行する
6. 仕様変更があれば既存ファイルを直接更新する（ファイル内の「最終更新」を変更する）

---

## チェックリスト

要件定義ドキュメントが以下を満たしているか確認する。

- [ ] 解決したい課題・背景が明記されている
- [ ] ゴール（このシステムで達成したいこと）が明記されている
- [ ] スコープ（対象範囲・対象外）が明記されている
- [ ] ステークホルダーが列挙されている
- [ ] 機能要件が列挙されている（ユーザーストーリー or 機能一覧）
- [ ] 非機能要件が定義されている（パフォーマンス・セキュリティ・可用性 etc.）
- [ ] 制約条件（技術・予算・期間・法的）が明記されている
- [ ] 用語定義が記載されている（ドメイン固有用語）
- [ ] リスクと対策が記載されている
- [ ] シャビの承認を得ている
