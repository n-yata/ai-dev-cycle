# フェーズ① 基本設計（basic-design）

詳細設計に入る前に、システム全体の方針・技術スタック・構成を固めるフェーズです。  
「何を作るか・どう作るか」をチーム全員が共通認識を持った状態で詳細設計へ移行することを目的とします。

---

## このフェーズの目的

- 技術スタック・アーキテクチャの合意形成
- システム全体構成の可視化
- 詳細設計・実装フェーズへの入力情報の整理

---

## 成果物一覧

| # | 成果物 | テンプレート | 単位 | 説明 |
|---|--------|-----------|------|------|
| 1 | 基本設計書 | [template.md](template.md) | 機能/アプリごと | 技術スタック・アーキテクチャ・API概要・データモデル概要・実装フェーズ計画 |

### ファイル命名規則

設計書は継続的に更新するため、ファイル名に日付は付けない。

```
docs/artifacts/basic-design/{機能名またはアプリ名}.md
例: docs/artifacts/basic-design/todo-app.md
例: docs/artifacts/basic-design/user-auth.md
```

---

## ディレクトリ構成

```
rules/basic-design/           # ルール・テンプレート（ここ）
├── INSTRUCTIONS.md
└── template.md

artifacts/basic-design/       # 成果物の出力先
└── {機能名}.md
```

---

## 基本設計ドキュメントの作成手順

1. `docs/rules/basic-design/template.md` をコピーする
2. `docs/artifacts/basic-design/` 配下に命名規則に従って保存する
3. テンプレートの各セクションを埋める
4. チームレビューを実施する
5. 承認後、詳細設計フェーズへ移行する
6. 仕様変更があれば既存ファイルを直接更新する（ファイル内の「最終更新」を変更する）

---

## チェックリスト

基本設計ドキュメントが以下を満たしているか確認する。

- [ ] 機能の目的・背景が明記されている
- [ ] 技術スタックとその選定理由が記載されている
- [ ] 却下した技術選択肢とその理由が記載されている
- [ ] システム全体のディレクトリ構成が示されている
- [ ] データモデルの概要（エンティティ・主要フィールド）が示されている
- [ ] APIエンドポイントの一覧（概要レベル）が示されている
- [ ] フロントエンドのコンポーネント構成が示されている
- [ ] 環境変数の管理方針が明記されている（ハードコード禁止）
- [ ] 実装フェーズ計画が示されている
- [ ] レビュアーの承認を得ている
