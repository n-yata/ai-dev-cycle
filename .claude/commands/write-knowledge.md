# ナレッジ書き込みコマンド

この会話で記録すべきナレッジを `docs/knowledge/` に書き込む。
自動トリガー（[KNOWLEDGE_TRIGGER]）と同じ処理を、任意のタイミングで手動実行する。

---

## 実行手順

### Step 1: 記録すべきナレッジの判断

この会話を振り返り、以下のいずれかに該当するものを抽出する:

| 判断基準 | カテゴリ | 格納先 |
|---------|---------|--------|
| 設計判断・技術スタック選定・却下した選択肢 | design-decisions | `docs/knowledge/design-decisions/` |
| コードレビューの指摘パターン | review-findings | `docs/knowledge/review-findings/` |
| バグパターン・効果的なテスト手法 | test-patterns | `docs/knowledge/test-patterns/` |
| フェーズ振り返り・教訓 | lessons-learned | `docs/knowledge/lessons-learned/` |

**記録しない（スキップ）**:
- 公式ドキュメントに書いてあること
- コードを読めば自明な内容
- 一時的な作業メモ・現在の会話スコープのみに有効な内容

記録すべきナレッジが1つもない場合は「記録すべきナレッジはありませんでした」と報告して終了する。

### Step 2: 各ナレッジのファイル作成

テンプレート `docs/rules/knowledge/template.md` を参照し、ナレッジファイルを作成する。

**ファイル名**: `YYYYMMDD_短いタイトル.md`（今日の日付を使用）  
**格納先**: カテゴリに対応するサブディレクトリ

テンプレートの全セクションを埋める必要はない。以下は**必須**:
- メタデータ表（記録日、フェーズ、カテゴリ、対象領域、重要度）
- 概要
- 事実（何が起きたか / 何を決めたか）
- 理由・分析
- 次回への示唆（Do / Don't）

### Step 3: 結果報告

以下の形式で報告する:

```
## ナレッジ書き込み結果

| ファイル | カテゴリ | 概要 |
|---------|---------|------|
| ファイルパス | カテゴリ名 | 内容の一言要約 |

### スキップした内容
- 内容（理由）
```

---

## 注意事項

- `docs/knowledge/` 直下には絶対に置かない（必ずカテゴリサブディレクトリに格納）
- 複数のカテゴリに該当するナレッジは、最も主要なカテゴリ1つを選ぶ
- 同じ内容のナレッジが既に存在する場合は重複して作成しない（既存ファイルを更新する）
- ナレッジをルールに反映したい場合は `/reflect-knowledge` を実行する
